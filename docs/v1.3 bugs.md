current working-tree version is `1.4.0`, but this remains the v1.3 finding list.

## V13-001: AWS telemetry topic was wrong

- **Symptom:** Device published to legacy `dt/propane-tank/<device-id>/telemetry`.
- **Expected:** Publish to `logi4wifi/device/<device-id>` using the full Thing UUID.
- **Impact:** Cloud rules/subscribers on the LOGI-4W topic miss all telemetry.
- **Status:** Fixed in `main/MQTT/AwsIotConfig.cpp`.
- **Validate:** Subscribe to `logi4wifi/device/+` and confirm publish after provisioning and normal post.

## V13-002: MQTT telemetry JSON did not match schema

- **Symptom:** Payload used legacy/verbose fields, then later correct keys with placeholder values such as `bat: 0`, `raw: 0`, and early `sch: "3.1.1"`.
- **Expected:** Short-form LOGI-4W keys: `dev`, `dts`, `sch`, `ver`, `lsq`, `bat`, `ful`, `amb`, `sol`, `chg`, `rst`, `lat`, `lon`, `alt`, `gsq`, `err`, `raw`, `supv`, `psch`, `fdt`, `fpd`, `pdt`, `wto`, `bleadv`.
- **Impact:** Cloud parser/dashboard cannot reliably ingest telemetry.
- **Status:** Fixed in `main/MQTT/AwsIotClient.cpp`; payload now emits Rev B short-form JSON from `TelemetryContext` and `LogiSensorData`.
- **Validate:** Capture MQTT packet, confirm topic, exact keys, no legacy keys, correct schema version, and two-decimal `raw`/`supv`.

## V13-003: Shadow reported/parser shape was wrong

- **Symptom:** Reported shadow included old demo field `welcome: "aws-iot"`, empty schedule arrays, and legacy `lte_timeout`.
- **Expected:** Device accepts desired fields including comma-separated `post_schedule`, `fill_dwell_time`, `wifi_timeout`, `fill_alarm_delta`, `post_dwell_time`, `mqtt_scheduled_post`, `event_posts`, `event_thresholds_pct`, `sensor_sample_rate`, `acquire_gps`, `mqtt_timeout`, and `ble_adv_time`.
- **Impact:** Cloud settings were not reliably applied or echoed in the expected LOGI-4W shape.
- **Status:** Fixed in `main/MQTT/ShadowParser.cpp`; supports string schedule, trims entries, parses new desired fields, and reports clean shape.
- **Validate:** Push desired shadow updates and confirm logs, reported state, and second telemetry post reflect the changes.

## V13-004: DTS was not gated on NTP

- **Symptom:** Telemetry used epoch timestamps such as `1970-01-01T00:02:56.000`.
- **Expected:** After Wi-Fi connects, device gets UTC time from NTP before TLS/MQTT and before populating `dts`.
- **Impact:** TLS validation can fail and cloud event ordering is unusable.
- **Status:** Fixed in application flow.
- **Validate:** Fresh provision, confirm NTP sync happens before MQTT connect and first publish has current UTC `dts`.

## V13-005: Activation/provisioning workflow was incomplete

- **Symptom:** After successful provisioning, the device did not consistently run the required activation cycle.
- **Expected:** Wi-Fi + NTP, immediate telemetry post, shadow sync, jobs/OTA check, GPS/sensor acquisition, second telemetry post, green LED for rest of cycle, then sleep.
- **Impact:** Cloud can miss activation, shadow settings may not apply before next cycle, OTA jobs may be delayed, and GPS/sensor values may never get corrected.
- **Status:** Fixed in provisioning/posting state machines.
- **Validate:** Provision clean unit while subscribed to telemetry, shadow, and jobs topics. Confirm two telemetry publishes with shadow/jobs activity between them.

## V13-006: Startup LED cycled through wrong colors

- **Symptom:** Device showed yellow/red/blue during startup/provisioning.
- **Expected:** Go straight to blue provisioning indication.
- **Impact:** Startup looked like warning/error states before an actual failure.
- **Status:** Fixed; bench observed blue flashing.
- **Validate:** Power-cycle and confirm no yellow/red sequence before blue.

## V13-007: RGB LED / I2C initialization could fail

- **Symptom:** Logs showed LED/I2C initialization failures even though the LED hardware works.
- **Expected:** LED initialization is critical and should recover/retry if the I2C bus is bad.
- **Impact:** Device may appear dead or provide no provisioning/activation status indication.
- **Status:** Fixed in LED/hardware driver paths with required LED init, I2C recovery/retry, and SPS held low during startup. Measurement paths enable SPS only while sampling.
- **Validate:** Repeated battery power-cycle/deep-sleep soak with no LED init errors and successful later sensor reads.

## V13-008: Provisioning PoP was dynamic

- **Symptom:** Proof of Possession varied per device.
- **Expected:** Static PoP: `F10974B8`.
- **Impact:** App/provisioning flow cannot rely on a known product PoP.
- **Status:** Fixed.
- **Validate:** Provision multiple units with static PoP.

## V13-009: App partition was effectively full

- **Symptom:** App image was ~99% of partition: `0x1ad5b0` used in `0x1b0000`, only `0x2a50` free.
- **Expected:** About 80% used or less to preserve OTA/feature margin.
- **Impact:** Tiny changes could break build/OTA margin.
- **Status:** Fixed in `sdkconfig.defaults`. Latest build is `0x1528f0` used with `0x5d710` free, about 78% used / 22% free.
- **Changes:** Size optimization, assertions off, no full x509 cert bundle, no MQTT websockets, trimmed BLE roles/services, disabled SoftAP/Enterprise/EAP/WPA3 extras/DHCP server/IPv6.
- **Tradeoff:** Assumes BLE provisioning and STA-mode Wi-Fi; SoftAP provisioning, Enterprise Wi-Fi, IPv6, and WPA3-only networks are not supported.
- **Validate:** Clean build from regenerated `sdkconfig` stays near 78% used.

## V13-010: Telemetry values were zero or sentinel values

- **Symptom:** Payload had values like `bat: 0`, `ful: 0`, `amb: 16513`, `sol: 0`, `raw: 0`, `supv: 0`, and GPS zeros.
- **Expected:** Real values from sensors/network/config; GPS may be `0` only until fix is available.
- **Impact:** JSON is valid but operationally useless for dashboards, alerts, and analytics.
- **Status:** Partially fixed. Telemetry now pulls from `TelemetryContext`, `LogiSensorData`, reset counter, Wi-Fi RSSI, local settings, and shadow-applied settings. SHT4x out-of-range reads are rejected with fallback to last valid ambient sample.
- **Validate:** Compare MQTT payload to serial logs and known bench measurements after a full sensor/GPS post cycle.

## V13-011: Reset counter missing from telemetry

- **Symptom:** Required `rst` key was absent.
- **Expected:** Payload includes `rst` reset counter.
- **Impact:** Cloud loses reboot-loop/brownout diagnostic signal.
- **Status:** Fixed with `main/logi/ResetCounter.*` and `AwsIotClient` payload update.
- **Validate:** Reset/power-cycle several times and confirm `rst` behavior matches intended semantics.

## V13-012: Build/provision tooling paths were machine-specific

- **Symptom:** Local provisioning needed correct ESP-IDF/cert paths and PowerShell blocked unsigned script execution.
- **Expected:** Bench machine uses ESP-IDF `C:\Espressif\esp-idf-v5.5.2` and certs under `C:\LOGI-W\LOGI4-W-Certs\ed60838a-24cb-42a0-b358-670984739837`.
- **Impact:** Easy to build/provision against wrong IDF or cert folder.
- **Status:** Fixed for this local bench machine.
- **Validate:** Run script with process-scope execution bypass, flash/provision on `COM9`, and confirm cert/device ID load in logs.

## V13-013: Activation cycle waits for an open COM port

- **Symptom:** Current device does not begin its activation cycle until a PC opens a valid COM port for logging. The blue provisioning/activation LED does not turn on unless the USB serial COM port is opened.
- **Expected:** Activation must start on power/battery without any PC, terminal, serial monitor, or logging consumer attached.
- **Impact:** This is a production blocker. Field units will not have a COM port open, so the device may look dead and never enter the visible activation/provisioning flow.
- **Status:** Fixed in `main/StateMachines/ProvisioningStateMachine.cpp`. Root cause was NOT a logging/console wait: the provisioning window enabled auto **light sleep**, which hard-hangs the device and triggers an interrupt-watchdog reset (INT_WDT, `reset_reason=5`) - a free-running LED I2C write lands across a light-sleep entry/exit on the marginal bus and stalls with interrupts disabled. It only appeared to "wait for a COM port" because the USJ keep-awake lock (`CONFIG_USJ_NO_AUTO_LS_ON_CONNECTION`) holds the device out of light sleep while a host has the port open. Fix: keep the provisioning window awake (do not allow light sleep) - it saved ~no power anyway since Wi-Fi STA is up the whole window. Bench-validated 2026-06-17: power-cycled on bench supply with no serial monitor, the device enters provisioning, blue LED on, advertising `MyPropane-ed60` (confirmed by passive BLE scan with the COM port free).
- **Validate:** Reproduce with battery-only/no serial monitor, then with serial monitor open. Fix is accepted only when blue LED and activation cycle start without opening COM.
