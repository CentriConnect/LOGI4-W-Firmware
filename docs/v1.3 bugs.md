# LOGI-4W v1.3 Bug List

This document captures the bugs and release blockers identified while bench testing the v1.3
firmware against the LOGI-4W Rev B firmware/update expectations.

Current working-tree firmware version is `1.4.0`, but these findings are retained as the v1.3 bug
record because they were discovered during the v1.3 bring-up.

## V13-001: AWS telemetry topic was wrong

**Observed behavior**

The device was still publishing telemetry to the legacy propane tank topic:

```text
dt/propane-tank/<device-id>/telemetry
```

**Expected behavior**

Telemetry must publish to:

```text
logi4wifi/device/<device-id>
```

where `<device-id>` is the full Thing UUID, for example:

```text
logi4wifi/device/ed60838a-24cb-42a0-b358-670984739837
```

**Impact**

Cloud rules/subscribers listening to the LOGI-4W topic would never receive device telemetry.

**Fix**

`main/MQTT/AwsIotConfig.cpp` now builds the telemetry topic as `logi4wifi/device/%s`.

**Validation needed**

Subscribe in AWS IoT MQTT test client to `logi4wifi/device/+` and confirm the unit publishes on the
new topic after provisioning and during normal post cycles.

## V13-002: MQTT telemetry JSON did not match required schema

**Observed behavior**

The posted packet originally used legacy/verbose fields and then later posted the right keys with
incorrect or placeholder values:

```json
{
  "dev": "ed60838a-24cb-42a0-b358-670984739837",
  "dts": "1970-01-01T00:02:56.000",
  "sch": "3.1.1",
  "ver": "4.1,1.2,3.1",
  "lsq": -53,
  "bat": 0,
  "ful": 0,
  "amb": 0,
  "sol": 0,
  "chg": 0,
  "raw": 0,
  "supv": 0
}
```

**Expected behavior**

Telemetry must use the LOGI-4W short-form keys:

```json
{
  "dev": "...",
  "dts": "2026-06-16T17:00:19.000",
  "sch": "4.1.1",
  "ver": "4.1,0.0,4.1.1",
  "lsq": -86,
  "bat": 4.03,
  "ful": 53,
  "amb": 54,
  "sol": 2.38,
  "chg": 98,
  "rst": 0,
  "lat": 32.803733825683594,
  "lon": -96.873382568359375,
  "alt": 129.67637634277344,
  "gsq": "34,0",
  "err": "",
  "raw": 2.16,
  "supv": 4.02,
  "psch": "10:00;FF, 22:00;FF, 06:00;FF, 00:00;00, 00:00;00, 00:00;00, 00:00;00, 00:00;00",
  "fdt": 900,
  "fpd": 10,
  "pdt": 120,
  "wto": 120,
  "bleadv": 120
}
```

**Impact**

The cloud parser and dashboard cannot reliably ingest the packet if fields are missing, renamed,
or filled with placeholders.

**Fix**

`main/MQTT/AwsIotClient.cpp` now emits the Rev B short-form payload from `TelemetryContext` and
`LogiSensorData`, including `rst`, `wto`, and `bleadv`.

**Validation needed**

Capture the MQTT packet from the AWS IoT MQTT test client and verify:

- Topic is `logi4wifi/device/<device-id>`.
- No legacy fields are present.
- `raw` and `supv` are reported to two decimals.
- `sch` is the MQTT schema version, not the MQTT protocol version.
- Config echo fields match local/shadow-applied settings.

## V13-003: Shadow reported/parser shape was wrong

**Observed behavior**

Reported shadow included the old demo-looking `welcome: "aws-iot"` field and represented
`post_schedule` as an array of empty strings:

```json
{
  "reported": {
    "welcome": "aws-iot",
    "post_schedule": ["", "", "", "", "", "", "", ""],
    "fill_dwell_time": 0,
    "wifi_timeout": 0,
    "lte_timeout": 0,
    "fill_alarm_delta": 0,
    "post_dwell_time": 0,
    "ble_adv_time": 0
  }
}
```

**Expected behavior**

The device must accept the desired-state shape used by the cloud:

```json
{
  "state": {
    "desired": {
      "post_schedule": "10:00;FF,16:00;FF,22:00;FF,00:00;00,00:00;00,00:00;00,00:00;00,00:00;00",
      "fill_dwell_time": 900,
      "wifi_timeout": 1000,
      "fill_alarm_delta": 20,
      "post_dwell_time": 120,
      "mqtt_scheduled_post": "19:00;88",
      "event_posts": true,
      "event_thresholds_pct": "5,10,20,30,40,50",
      "sensor_sample_rate": 10,
      "acquire_gps": false,
      "mqtt_timeout": 120,
      "ble_adv_time": 120
    }
  }
}
```

**Impact**

Device settings from AWS shadow were not reliably applied to memory/NVS, and the reported state did
not match the expected LOGI-4W cloud contract.

**Fix**

`main/MQTT/ShadowParser.cpp` now accepts comma-separated `post_schedule`, retains support for array
format, trims entries, and parses the additional desired fields. Reported shadow output now writes
`post_schedule` as a string and omits the old AWS demo field.

**Validation needed**

Push desired shadow updates one field at a time and confirm:

- Device logs show the field was parsed/applied.
- Second telemetry post echoes the updated values.
- Reported shadow no longer includes `welcome` or `lte_timeout`.

## V13-004: DTS was not gated on NTP

**Observed behavior**

Telemetry used a Unix epoch timestamp such as:

```text
1970-01-01T00:02:56.000
```

**Expected behavior**

After Wi-Fi connects, the device must obtain a valid UTC timestamp from NTP before TLS/MQTT work.
TLS certificate validation and telemetry `dts` both depend on valid time.

**Impact**

Invalid time can break TLS validation and produces unusable telemetry ordering in the cloud.

**Fix**

The application flow now syncs time after Wi-Fi connect and before AWS IoT/TLS/MQTT operations.

**Validation needed**

Boot with erased NVS/provision fresh, then confirm logs show NTP sync before MQTT connect and the
first telemetry packet has a current UTC `dts`.

## V13-005: First successful provisioning workflow was incomplete

**Observed behavior**

After successful Wi-Fi provisioning and AWS connection, the device did not consistently run the
required activation sequence.

**Expected behavior**

After Wi-Fi connects and NTP succeeds, the device should:

1. Immediately post telemetry with the values available at that moment.
2. Get/apply the device shadow settings into memory/NVS.
3. Check AWS IoT jobs/OTA and run OTA if a job is available.
4. Acquire GPS/sensor values and post telemetry again on MQTT port 8883.
5. Finish the activation cycle with green LED blinking, then sleep.

**Impact**

The cloud may miss the first activation event, shadow defaults may not be applied before the next
cycle, OTA jobs may not be picked up promptly, and the first telemetry packet may lack GPS/sensor
updates without a later correction.

**Fix**

Provisioning and posting state machines were updated for the post -> jobs/shadow -> post workflow.

**Validation needed**

Provision a clean unit while subscribed to `logi4wifi/device/+` and AWS jobs/shadow topics. Confirm
two telemetry publishes occur in the expected order, with jobs/shadow traffic between them.

## V13-006: Startup LED cycled through the wrong colors

**Observed behavior**

On boot/provisioning, the LED showed a yellow/red/blue startup sequence. The requested product
behavior is to go directly to blue during provisioning.

**Expected behavior**

The provisioning indication should go straight to blue and avoid the confusing multi-color startup
cycle.

**Impact**

The old sequence looked like warning/error states before the device had actually failed.

**Fix**

Startup LED behavior was simplified so provisioning goes straight to blue. A bench run confirmed
the blue LED is flashing.

**Validation needed**

Power-cycle and start provisioning. Confirm no yellow/red startup sequence occurs before blue.

## V13-007: RGB LED / I2C initialization could fail

**Observed behavior**

Logs showed LED/I2C initialization failures even though the LED hardware was known to work. This is
critical because LED state is a required product signal during provisioning and activation.

**Expected behavior**

RGB LED initialization should be reliable. If the shared I2C bus is in a bad state, firmware should
recover/retry rather than silently continue with an unusable LED.

**Impact**

Users could see no product status indication, or the device could appear dead during provisioning.

**Fix**

The LED path now treats RGB LED initialization as required, adds I2C recovery/retry before giving
up, and avoids raising SPS during startup because SPS was interfering with the LED I2C device on
the rev-4.1 bench unit. Measurement paths enable SPS only while sampling.

**Validation needed**

Run a soak of repeated battery power cycles and deep-sleep wake cycles. Confirm no LED init errors,
blue provisioning indication works, and later sensor sampling still succeeds.

## V13-008: Provisioning PoP was dynamic

**Observed behavior**

The Proof of Possession was being generated dynamically/per-device.

**Expected behavior**

The product requirement is a static PoP.

**Impact**

The phone/app provisioning flow cannot use a consistent known PoP if each device changes it.

**Fix**

Provisioning now uses the static PoP:

```text
F10974B8
```

**Validation needed**

Provision with the app using the static PoP and confirm the device accepts it across multiple units.

## V13-009: App partition was effectively full

**Observed behavior**

The app partition was at approximately 99% capacity:

```text
LOGI4W.bin binary size 0x1ad5b0
Smallest app partition 0x1b0000
free 0x2a50
```

**Expected behavior**

The firmware image should be at or below roughly 80% used to leave OTA and feature-growth margin.

**Impact**

At 99%, small code changes could break the build and OTA margin was too thin for production.

**Fix**

`sdkconfig.defaults` now carries the production footprint settings:

- Size optimization instead of debug optimization.
- Assertions disabled for release footprint.
- ESP-IDF full x509 certificate bundle disabled.
- MQTT websocket transports disabled.
- TLS server-side features disabled where not needed.
- NimBLE trimmed to BLE provisioning peripheral/server roles.
- Unused NimBLE sample services disabled.
- Wi-Fi SoftAP, Enterprise/EAP, WPA3 extras, DHCP server, and IPv6 disabled.

**Validated result**

Latest build:

```text
LOGI4W.bin binary size 0x1528f0
Smallest app partition 0x1b0000
free 0x5d710
```

That is approximately 78% used / 22% free.

**Tradeoff**

This assumes the product uses BLE provisioning and STA-mode Wi-Fi. SoftAP provisioning, Enterprise
Wi-Fi, IPv6, and WPA3-only networks are not supported with this footprint configuration.

## V13-010: Telemetry values were still zero or sentinel values

**Observed behavior**

After schema cleanup, some values were still clearly wrong:

```json
{
  "bat": 0,
  "ful": 0,
  "amb": 16513,
  "sol": 0,
  "chg": 0,
  "lat": 0,
  "lon": 0,
  "alt": 0,
  "raw": 0,
  "supv": 0
}
```

**Expected behavior**

Telemetry values should come from real sensor/network/config sources:

- `bat`: battery voltage.
- `ful`: tank/fuel level.
- `amb`: ambient temperature.
- `sol`: solar voltage.
- `chg`: charger/charge status.
- `lat`, `lon`, `alt`: GPS if available, otherwise `0`.
- `raw`: raw sensor signal voltage, two decimals.
- `supv`: sensor supply voltage, two decimals.
- `psch`, `fdt`, `fpd`, `pdt`, `wto`, `bleadv`: current local/shadow-applied config.

**Impact**

The cloud receives syntactically valid JSON but cannot use the data for dashboards, alerts, or
analytics if the values are stale, zero, or sentinel readings.

**Fix**

Telemetry now pulls through `TelemetryContext`, `LogiSensorData`, reset counter, Wi-Fi RSSI, local
device settings, and shadow-applied settings. Sensor reads now reject out-of-range SHT4x values and
fall back to the last valid ambient sample when available.

**Validation needed**

Run a full post cycle after sensor sampling and compare MQTT payload against serial logs and known
bench measurements. GPS fields may remain `0` until fix acquisition succeeds.

## V13-011: Reset counter missing from payload

**Observed behavior**

The required `rst` key was not present in the initial LOGI-4W JSON.

**Expected behavior**

Telemetry must include:

```json
"rst": 0
```

where the value is the device reset counter.

**Impact**

Cloud diagnostics lose an important signal for reboot loops, brownouts, and repeated activation
attempts.

**Fix**

`main/logi/ResetCounter.*` was added and `AwsIotClient` includes `rst` in the telemetry payload.

**Validation needed**

Power-cycle/reset the unit multiple times and confirm `rst` increments or resets according to the
intended reset-counter semantics.

## V13-012: Build/provision tooling paths were not set up for this bench machine

**Observed behavior**

The local build/provision script needed machine-specific ESP-IDF and certificate paths. PowerShell
also blocked script execution due to execution policy:

```text
build-and-provision.ps1 cannot be loaded. The file is not digitally signed.
```

**Expected behavior**

The script should work on this bench machine with:

```text
ESP-IDF: C:\Espressif\esp-idf-v5.5.2
Cert directory: C:\LOGI-W\LOGI4-W-Certs\ed60838a-24cb-42a0-b358-670984739837
```

**Impact**

Factory build/provision steps are fragile and easy to run against the wrong ESP-IDF install or
wrong certificate folder.

**Fix**

The local build/provision script was updated for this machine and a clean build was run.

**Validation needed**

Run the full build/provision script from a PowerShell session launched with:

```powershell
Set-ExecutionPolicy -Scope Process -ExecutionPolicy Bypass
```

then flash/provision on `COM9` and confirm cert/device ID load in logs.
